## 1. 产品背景与目标

### 1.1 背景

企业在推进「岗位标准化」工作中，涉及岗位说明书、任职资格、绩效考核、SOP 等多类文档的起草与优化，人力资源团队耗费大量时间在重复性写作与格式规范中。本产品旨在以 AI 教练形式，让员工和 HR 可通过上传现有文档或直接提问，快速获得专业的标准化建议与改写内容。

### 1.2 产品目标

| 目标维度 | 具体指标 |
|---------|---------|
| 效率提升 | 单份标准化文档生成时间从平均 4 小时降至 30 分钟以内 |
| 专业度 | 输出内容符合企业人力资源专业规范，可直接使用或微调 |
| 易用性 | 无需培训，上传文件即可获得针对性指导 |
| 一致性 | 同类岗位文档风格与结构保持一致 |

### 1.3 核心用户

- **HR 专员 / BP**：负责岗位体系建设，需要批量产出规范文档
- **部门管理者**：需要为本团队岗位撰写描述和考核标准
- **组织发展 (OD) 顾问**：做岗位诊断和体系优化

## 3. 路由 Agent 设计

### 3.1 路由决策流程

```
用户输入
    │
    ├─── 有文件上传？
    │         │
    │         ├─ YES → 文件内容解析
    │         │              │
    │         │              └─ 关键词/结构特征匹配
    │         │                        │
    │         │                        ├─ 匹配成功 → 激活对应 Agent，更新 currentAgent
    │         │                        │
    │         │                        └─ 匹配失败（置信度 < 0.6）
    │         │                                    │
    │         │                                    └─ 询问用户确认文档类型
    │         │
    │         └─ NO → currentAgent 是否存在？
    │                        │
    │                        ├─ YES → 复用 currentAgent 继续对话
    │                        │
    │                        └─ NO → 激活 Agent 8（通用问答）
    │
    └─ 输出响应
```

### 3.2 路由 Agent 系统提示词

```
SYSTEM:
你是一个岗位标准化文档路由分析器。当用户上传文档时，你需要分析文档内容，
识别其属于以下哪种类型，并输出结构化的 JSON 路由决策。

文档类型映射：
- "jd"：岗位说明书、职位描述、岗位职责
- "qualification"：任职资格标准、资质要求
- "performance"：绩效考核标准、KPI/OKR 表格
- "training"：培训发展方案、学习地图、IDP
- "sop"：SOP、作业指导书、操作流程
- "compensation"：薪酬等级标准、薪级表
- "competency"：胜任力模型、素质模型
- "general"：无法识别或自由问答

分析时关注：
1. 文档标题中的关键词
2. 文档结构（有无"岗位职责""任职要求""KPI 指标"等典型 Section）
3. 内容语义（描述工作内容 vs 描述能力要求 vs 描述指标体系）

输出格式（仅 JSON，不输出任何其他内容）：
{
  "agent_type": "<类型英文标识>",
  "confidence": <0.0-1.0>,
  "reasoning": "<简短的识别依据，10-30字>",
  "doc_title": "<从文档中提取的文档名称或岗位名称>",
  "key_fields": ["<识别到的关键字段1>", "<关键字段2>"]
}
```

### 3.3 路由置信度策略

| 置信度范围 | 处理策略 |
|-----------|---------|
| ≥ 0.85 | 直接激活对应 Agent，无需确认 |
| 0.60 ~ 0.84 | 激活对应 Agent，但在首条回复中说明识别结果并请用户确认 |
| < 0.60 | 展示 top-2 候选，请用户手动选择 Agent 类型 |

## 4. 八类 Agent 详细设计

### 4.1 Agent 1 — 岗位说明书教练

**定位**：帮助用户将粗糙的岗位描述改写为符合企业规范的标准化岗位说明书。

**核心能力**：
- 识别并补全缺失的标准要素（岗位基本信息、汇报关系、职责条目、考核维度）
- 将散乱的职责描述拆解为"动词 + 宾语 + 结果"结构
- 输出可直接使用的标准 Word 格式内容

**标准输出结构**：
```
1. 岗位基本信息（岗位名称/编码/所属部门/岗位等级/汇报关系）
2. 岗位定位（一句话描述岗位价值）
3. 主要职责（5-8 条，"动词+对象+目的"格式）
4. 工作关系（内部/外部协作对象）
5. 业绩衡量（3-5 个 KPI 方向）
6. 工作环境与特殊要求
```

---

### 4.2 Agent 2 — 任职资格标准教练

**定位**：帮助用户建立清晰、可评估的岗位任职资格体系。

**核心能力**：
- 从三个维度拆解资格：基础条件（学历/经验）、专业知识、通用能力
- 区分"必要条件"与"加分条件"
- 按岗位级别（P1-P5 或初/中/高）设计差异化要求

**标准输出结构**：
```
1. 基础任职条件（学历、工作年限、证书）
2. 专业知识要求（领域知识、工具技能、行业背景）
3. 核心能力要求（可拆解至行为描述级别）
4. 加分项
5. 禁止条件（如有）
```

---

### 4.3 Agent 3 — 绩效考核标准教练

**定位**：帮助用户设计科学、量化、公平的岗位绩效考核体系。

**核心能力**：
- 将工作职责转化为可量化的 KPI 指标
- 设计指标权重分配（建议原则：核心指标 ≥ 60%）
- 撰写评分规则（5 分制 / 百分制的档位描述）
- 区分结果指标和过程指标

**标准输出结构**：
```
1. 考核周期与适用范围
2. 指标体系（类别 / 指标名称 / 计算公式 / 数据来源 / 权重）
3. 各档位评分标准（超越/达标/基本达标/未达标的行为描述）
4. 考核流程说明
5. 结果应用说明
```

---

### 4.4 Agent 4 — 培训发展方案教练

**定位**：帮助用户为特定岗位设计系统性的培训路径与发展计划。

**核心能力**：
- 基于任职资格识别能力差距
- 设计分阶段（入职期/成长期/成熟期）的学习路径
- 推荐培训形式（课堂/在岗/导师制/外部学习）
- 生成 IDP（个人发展计划）模板

**标准输出结构**：
```
1. 岗位能力画像（目标能力清单）
2. 分阶段培训计划（时间轴 + 培训内容 + 形式 + 评估方式）
3. 关键资源清单（推荐课程/书籍/项目实践）
4. 发展里程碑与检验标准
```

---

### 4.5 Agent 5 — SOP 流程教练

**定位**：帮助用户将非正式的操作经验转化为标准化、可传承的 SOP 文档。

**核心能力**：
- 识别流程中的关键节点、决策点和风险点
- 为每个步骤补充"负责人/工具/输出物/时间要求"四要素
- 识别缺失的异常处理分支
- 输出可视化流程图描述（文字版 Flowchart）

**标准输出结构**：
```
1. SOP 基本信息（编号/版本/生效日期/适用范围）
2. 目的与范围
3. 术语定义
4. 职责分工（RACI 矩阵）
5. 操作步骤（步骤编号 / 操作内容 / 负责人 / 工具 / 输出物 / 注意事项）
6. 异常处理
7. 相关文件
8. 修订记录
```

---

### 4.6 Agent 6 — 薪酬等级标准教练

**定位**：帮助用户设计或优化岗位薪酬等级结构，确保内部公平性与外部竞争性。

**核心能力**：
- 基于岗位评估结果建议薪酬级别对应关系
- 设计宽带薪酬结构（薪资带宽 Min/Mid/Max）
- 提供薪酬调整规则与晋升联动逻辑
- 市场对标建议（基于岗位层级通用参考区间）

**标准输出结构**：
```
1. 薪酬体系说明（定薪原则/适用范围）
2. 岗位等级与薪酬级别对应表
3. 各级薪资带（Min / P25 / Midpoint / P75 / Max）
4. 薪酬调整规则（定期调薪/晋升调薪/特殊调薪）
5. 绩效与薪酬联动说明
```

---

### 4.7 Agent 7 — 胜任力模型教练

**定位**：帮助用户构建基于冰山模型的岗位/序列胜任力模型，为人才识别与发展提供标准。

**核心能力**：
- 从行为事件访谈（BEI）素材中提炼胜任力维度
- 将能力维度拆解为可观测的行为锚点
- 区分通用胜任力和专业胜任力
- 按能力等级（1-5 级）描述差异化行为表现

**标准输出结构**：
```
1. 模型概述（适用序列/岗位层级/能力维度总览）
2. 通用胜任力模块（每项含：定义 / 重要性 / 1-5 级行为描述）
3. 专业胜任力模块（同上）
4. 各级别岗位胜任力要求矩阵
5. 应用说明（选人/育人/评人场景）
```

---

### 4.8 Agent 8 — 通用问答教练

**定位**：当用户无文件上传或提出跨领域问题时，作为全能的岗位标准化顾问答疑解惑。

**核心能力**：
- 解答岗位标准化相关的概念问题
- 提供方法论建议（如"如何开展岗位梳理项目"）
- 给出岗位类文档的最佳实践
- 在用户描述需求后，引导其上传对应文件进入专属 Agent

## 5. 提示词规范与模板

### 5.1 提示词通用规范

每个 Agent 的提示词遵循以下结构：

```
[SYSTEM]        → 角色定义、能力边界、输出规范、格式约束
[USER Turn 1]   → 预设的开场白或文件接收确认（预加载用）
[ASSISTANT Turn 1] → 预设的欢迎语 + 文档初步分析（预加载用）
[USER Turn N]   → 实际用户输入
[ASSISTANT Turn N] → 实际 AI 回复
```

### 5.2 各 Agent 完整提示词模板

#### Agent 1：岗位说明书教练

```yaml
# ===== SYSTEM =====
system: |
  你是一位资深的人力资源专家，专注于企业岗位说明书的标准化设计。
  你拥有 15 年以上企业 HR 体系建设经验，曾服务于多家 500 强企业的组织发展项目。
  
  【你的工作方式】
  1. 当用户上传岗位说明书或岗位相关文档时，先进行要素完整度诊断
  2. 以"教练对话"风格逐步引导，不一次性抛出所有问题
  3. 每次回复聚焦一个核心议题（如：职责条目、考核维度）
  4. 输出内容时，始终提供"原文对比 → 问题诊断 → 优化建议 → 参考示例"四段式结构
  
  【输出规范】
  - 职责条目格式：动词（主动式）+ 宾语 + 目的/结果，例："制定年度培训预算，确保培训投入产出比≥150%"
  - 每条职责控制在 30 字以内
  - 主要职责数量：5-8 条
  - 避免使用"负责"作为唯一动词，应使用"制定/推进/协调/审核/优化/监督"等动词变体
  
  【边界约束】
  - 仅处理岗位说明书相关话题
  - 如用户询问薪酬/绩效等其他模块，友好说明可切换至对应教练
  - 不生成虚假的岗位编码或公司信息，使用[待填写]占位

# ===== 预加载 Turn 1（文件已上传时自动触发）=====
user_preload: |
  [用户上传了文件：{{doc_title}}]

assistant_preload: |
  收到你上传的「{{doc_title}}」👋
  
  我已快速扫描了文档内容，进行了初步的**要素完整度诊断**：
  
  **✅ 已有要素**
  {{detected_elements}}
  
  **⚠️ 待完善要素**
  {{missing_elements}}

  你希望先从哪里开始？或者直接告诉我你最想改善的地方？

# ===== 持续对话示例（few-shot）=====
user_example_1: |
  帮我把职责部分改写一下，现在写的太口语化了

assistant_example_1: |
  好的，我来逐条分析你现有的职责描述，找出可以优化的点：

  ---

```

#### Agent 3：绩效考核标准教练（示例，其余 Agent 结构相同）

```yaml
system: |
  你是一位绩效管理专家，专注于 KPI 体系设计与绩效考核标准建立。
  你熟悉平衡计分卡（BSC）、OKR、360 度评估等主流绩效方法论。
  
  【核心工作原则】
  1. SMART 原则检验：所有指标必须满足 Specific/Measurable/Achievable/Relevant/Time-bound
  2. 权重设计原则：结果指标权重 ≥ 60%，过程/行为指标 ≤ 40%
  3. 数据可获得性：每个指标必须明确数据来源，避免无法核实的定性描述
  4. 上下对齐原则：岗位 KPI 需与部门/公司目标有明确传导逻辑
  
  【诊断框架】
  - 指标数量是否合理（建议 5-8 个核心指标）
  - 是否存在"只考核行为不考核结果"的问题
  - 权重分配是否反映岗位核心价值
  - 评分标准是否可操作、无歧义

user_preload: |
  [用户上传了绩效考核相关文件：{{doc_title}}]

assistant_preload: |
```

## 6. 预加载机制设计

### 6.1 预加载目标

用户上传文件后，系统在正式展示对话框前，已在后台完成：
1. 文件内容解析（PDF/Word/TXT）
2. 路由分类识别
3. 对应 Agent 的初始化（System Prompt 注入）
4. 首轮分析响应预生成（文档诊断结果）

用户看到的是"即时响应"体验，而非等待体验。

### 6.2 预加载触发时机

```
用户点击"上传"或拖入文件
        ↓
文件开始上传（后台并行启动）
        ├── 任务 A：文件内容提取（流式）
        ├── 任务 B：路由分析 API 调用（基于已提取片段）
        └── 任务 C：目标 Agent 系统提示词缓存到会话

上传完成（用户尚未提交）
        ↓
Route 结果 Ready → 预生成首轮 Assistant 响应（流式）
        ↓
用户按"发送"或系统自动展示分析结果
```

### 6.3 预加载数据结构

```typescript
interface PreloadContext {
  sessionId: string;
  agentType: AgentType;            // 路由结果
  confidence: number;              // 路由置信度
  docTitle: string;                // 提取的文档标题
  detectedElements: string[];      // 已识别要素
  missingElements: string[];       // 缺失要素
  keyInsights: string[];           // 3-5 条快速洞察
  preloadedResponse: string;       // 预生成的首轮回复（流式拼接）
  conversationHistory: Message[];  // 已包含 system + preload turns
}
```

### 6.4 预加载状态 UI

```
[上传中]  📎 正在读取「岗位说明书-产品经理.docx」...
[分析中]  🔍 正在识别文档类型...（进度条 0→40%）
[就绪]    ✅ 已识别为「岗位说明书」，正在为你准备分析报告...（进度条 40→100%）
[完成]    → 自动切换到对话界面，展示预生成的首轮分析
```

## 7. 状态管理设计

### 7.1 会话状态模型

```typescript
interface SessionState {
  // 当前激活的 Agent
  currentAgent: AgentType | null;
  
  // Agent 激活时间戳（用于超时判断）
  agentActivatedAt: number | null;
  
  // 是否已有文件上传过
  hasUploadedFile: boolean;
  
  // 对话历史（含 system prompt）
  conversationHistory: Message[];
  
  // 当前文档信息
  currentDocument: {
    fileName: string;
    docType: AgentType;
    uploadedAt: number;
    summary: DocumentSummary;
  } | null;
  
  // 路由决策记录（用于复盘/反馈）
  routingDecisions: RoutingDecision[];
}
```

### 7.2 Agent 切换规则

| 场景 | 行为 |
|------|------|
| 用户上传新文件（与当前文件同类型） | 维持当前 Agent，追加新文件内容到上下文 |
| 用户上传新文件（不同类型） | 提示"检测到新类型文档，是否切换到 XXX 教练？"，用户确认后切换 |
| 用户纯文字提问（无文件） | 维持 currentAgent，继续在当前 Agent 语境下回复 |
| 用户明确说"换个问题"/"我要问别的" | 提示可用的 Agent 类型，用户选择后切换 |
| 会话空闲 > 30 分钟 | 保持 Agent 状态，但在用户回来时发送简短的上下文提示 |

### 7.3 对话历史管理

为控制 Context Window 消耗，采用**滑动窗口 + 摘要**策略：

```
完整历史 > 20 轮时：
    保留 [system] + [前 2 轮（首次分析）] + [最近 10 轮] + [中间轮的摘要]

摘要生成提示词：
"请将以下对话历史压缩为 200 字以内的关键要点摘要，
保留：已确认的修改项、用户的核心需求、文档主要问题。"
```

## 8. 文件解析与识别机制

### 8.1 支持的文件类型

| 类型 | 格式 | 解析方式 | 优先级 |
|------|------|---------|-------|
| Word 文档 | .docx / .doc | 提取正文 + 标题层级 + 表格 | ⭐⭐⭐ |
| PDF | .pdf | 文本提取（含表格识别） | ⭐⭐⭐ |
| 纯文本 | .txt | 直接读取 | ⭐⭐ |
| Excel | .xlsx | 表格结构化提取 | ⭐⭐（适用于绩效/薪酬模块）|
| 图片 | .jpg/.png | OCR 识别（限清晰扫描件）| ⭐ |

### 8.2 关键词识别矩阵

| Agent 类型 | 强信号词（权重 3） | 中信号词（权重 2） | 弱信号词（权重 1） |
|-----------|----------------|----------------|----------------|
| jd | 岗位说明书、职位描述、岗位职责 | JD、工作内容、汇报关系 | 岗位、职位、部门 |
| qualification | 任职资格、资质要求 | 专业背景、学历要求、经验要求 | 条件、要求 |
| performance | 绩效考核、KPI、OKR | 考核指标、评分标准、目标 | 考核、评估、指标 |
| training | 培训计划、学习地图、IDP | 发展路径、能力提升、培训方案 | 培训、学习、发展 |
| sop | SOP、作业指导书、操作流程 | 操作步骤、工作规范、标准程序 | 流程、步骤、规范 |
| compensation | 薪酬等级、薪级表、工资带 | 定薪标准、薪资结构、薪酬体系 | 薪酬、工资、待遇 |
| competency | 胜任力模型、素质模型 | 能力维度、行为锚点、冰山模型 | 能力、素质、模型 |

**识别分数计算**：累加各类型匹配词的权重分，分数最高者为路由结果，归一化得 confidence。

## 9. 对话流程设计

### 9.1 首次进入流程（无文件）

```
用户打开应用
    ↓
Agent 8（通用）自动激活
    ↓
展示欢迎界面（含功能介绍 + 两种使用方式说明）
    ↓
用户可选：① 直接提问  ② 上传文件
```

### 9.2 文件上传流程（核心路径）

```
用户上传文件
    ↓
[后台] 并行：文件解析 + 路由分析 + Agent 预加载
    ↓
路由结果返回（< 5 秒）
    ↓
置信度 ≥ 0.85？
    ├─ YES → 自动切换 Agent，展示"已为你切换到 XXX 教练"提示条
    │         展示预加载的首轮分析（流式输出）
    │
    └─ NO  → 展示候选 Agent 卡片，用户点击选择
              用户选择后，切换 Agent，展示预加载内容
```

### 9.3 多轮对话流程（维持 Agent 状态）

```
用户继续发送文字消息
    ↓
检查是否有新文件？
    ├─ NO → 直接复用 currentAgent，将消息加入历史，调用 API
    └─ YES → 重新执行路由逻辑（见 9.2）

API 调用
    ↓
流式输出响应（stream: true）
    ↓
每轮结束后：保存对话历史、触发历史摘要检查（> 20 轮时压缩）
```

### 9.4 快捷操作流程

每次 Agent 回复后，根据当前上下文自动生成 2-3 个快捷追问按钮，例如：

- 岗位说明书教练：`[ 继续优化下一条职责 ]  [ 生成完整文档 ]  [ 检查汇报关系 ]`
- 绩效考核教练：`[ 帮我量化这个指标 ]  [ 设计评分档次 ]  [ 导出指标表格 ]`

### 11.3 提示词注入机制

```typescript
function buildMessages(
  agentType: AgentType,
  fileContent: string,
  conversationHistory: Message[]
): Message[] {
  const systemPrompt = AGENT_PROMPTS[agentType].system;
  const preloadContext = buildPreloadContext(fileContent, agentType);
  
  return [
    { role: "system", content: systemPrompt },
    // 预加载 turns（作为历史注入，模拟文档已分析状态）
    { role: "user", content: preloadContext.userPreload },
    { role: "assistant", content: preloadContext.assistantPreload },
    // 实际对话历史
    ...conversationHistory
  ];
}
```

### 11.4 关键性能优化

| 优化点 | 方案 | 预期收益 |
|-------|------|---------|
| System Prompt 缓存 | 使用 Claude API 的 prompt caching（cache_control） | 减少 Token 费用约 70% |
| 文件解析客户端化 | mammoth.js 在浏览器端解析 docx，无需上传服务器 | 减少服务器带宽 |
| 流式输出 | SSE 流式返回，首字显示时间 < 1 秒 | 提升感知速度 |
| 路由并发 | 文件上传同时并发路由分析，不等上传完成 | 节省 2-4 秒 |
| 快捷按钮预生成 | 主回复生成时，同步生成快捷追问建议 | 减少等待 |

## 12. 错误处理与边界情况

### 12.1 文件识别失败

```
情况：文档为空白、乱码、或无关内容（如合同、发票）
处理：展示友好提示 + 手动选择 Agent 的卡片组
提示语："这份文件我还没能识别出类型，请选择你想使用的教练模式："
```

### 12.2 文件过大

```
限制：单文件 ≤ 10MB；文本提取 ≤ 50,000 字（超出截取前 50,000 字）
处理：展示"文件较大，已提取前 XXX 字进行分析，如需分析完整文件建议分段上传"
```

### 12.3 API 调用失败

```
情况：网络错误 / Claude API 超时 / 限流
处理：
  - 显示错误提示 + 重试按钮
  - 自动重试 1 次（指数退避）
  - 2 次失败后，提示用户稍后再试
```

### 12.4 跨 Agent 话题漂移

```
情况：用户在绩效教练中问薪酬问题
处理：
  - Agent 先简要回答（保持对话流畅）
  - 在回复末尾提示："深度分析薪酬等级，可切换到【薪酬等级标准教练】→"
  - 用户点击链接可一键切换，携带当前上下文
```

### 12.5 敏感信息提醒

```
情况：用户文件包含员工真实姓名、薪资数据
处理：
  - 文件解析时检测敏感模式（身份证号、银行卡号）
  - 展示隐私提示："建议上传前对敏感信息脱敏（如将真实姓名替换为'张三'）"
  - 不主动拦截，用户知情后自行决定
```

## 附录 C：快捷追问按钮生成提示词

```
在你的主要回复生成完毕后，额外输出一个 JSON 数组（放在 <quick_actions> 标签内），
包含 2-3 个最相关的后续操作建议，格式如下：

<quick_actions>
[
  { "label": "生成完整文档", "prompt": "请基于以上分析，帮我生成完整的标准化岗位说明书" },
  { "label": "继续下一条职责", "prompt": "请继续分析第3条职责" }
]
</quick_actions>

前端解析此标签并渲染为可点击按钮，不在对话界面显示原始 JSON。
```
